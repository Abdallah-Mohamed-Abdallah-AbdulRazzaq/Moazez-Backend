## Audit Notes

The actual implementation was inspected before writing this handoff. The following details are the final source of truth for frontend integration:

- App media attachment download/preview routes use `:attachmentId`, not `:fileId`, in the nested app routes.
- The shared file download route is implemented as `/api/v1/files/:id/download`; frontend may treat `:id` as the file id.
- Parent and Student message/notification contracts use dual camelCase plus snake_case aliases where implemented.
- Teacher message/notification/announcement contracts are camelCase.
- Core Communication routes remain management/core routes. App clients should use Parent, Student, and Teacher app routes.
- Message mentions are still deferred. `message_mention` is not generated by Track A runtime.
- Teacher announcement attachment linking is deferred. Teacher announcement payloads include `attachmentsCount`.
- Direct conversation reuse is service-level best effort using active two-participant DIRECT conversations; there is no database-level unordered direct pair unique constraint.

## Auth And Authorization Summary

Parent:

- Current actor is derived through Parent App access services.
- Message conversations are limited to parent-accessible conversations and linked-child context.
- Contact discovery returns teachers allocated to linked child classrooms only.
- Conversation creation accepts only a resolved `teacher:<userId>` contact from the parent contact read model.
- Attachment download/preview requires parent app access, visible conversation, matching message, matching attachment, and same-school non-deleted file.
- Notification center and preferences are scoped to the current parent user.

Student:

- Current actor is derived through Student App access services.
- Message conversations are limited to current active student/enrollment and participant scope.
- Contact discovery returns teachers allocated to the current classroom only.
- Conversation creation accepts only a resolved `teacher:<userId>` contact from the student contact read model.
- Attachment download/preview requires student app access, visible conversation, matching message, matching attachment, and same-school non-deleted file.
- Notification center and preferences are scoped to the current student user.

Teacher:

- Current actor is derived through Teacher App access services.
- Message conversations are limited to participant/teacher-accessible conversations.
- Contact discovery returns students in allocated classrooms and linked guardians for those students.
- Conversation creation accepts only resolved `student:<studentId>` or `guardian:<guardianId>` contacts from the teacher contact read model.
- Teacher announcement mutations require owned teacher-created announcement records and authorized classroom allocations.
- Attachment download/preview requires teacher app access, visible conversation, matching message, matching attachment, and same-school non-deleted file.
- Notification center and preferences are scoped to the current teacher user.

Core Communication:

- Core routes require current-school scope and route permissions.
- Message notification generation excludes the sender, inactive/removed/blocked/non-readable participants, and muted participants where existing participant status indicates notification suppression.
- Recipient notification preferences are applied after eligibility and before persistence/realtime.
- Notification realtime events target recipient user rooms only.
- Direct conversation creation never accepts arbitrary participant arrays from app routes.

## Conversation Contract

App-facing conversation list/detail responses are enriched and app-safe.

Key fields:

```json
{
  "conversationId": "uuid",
  "conversation_id": "uuid",
  "type": "direct",
  "status": "active",
  "isGroup": false,
  "is_group": false,
  "participantsCount": 2,
  "participants_count": 2,
  "unreadCount": 0,
  "unread_count": 0,
  "lastMessage": null,
  "lastMessageReadCount": 0,
  "last_message_read_count": 0
}
```

Rules:

- Parent/Student include snake_case aliases where implemented.
- Teacher uses camelCase.
- `unreadCount` is actor-specific and excludes messages sent by the current actor.
- `participantsCount` uses active + muted participants and includes the sender/current actor where active or muted.
- `lastMessageReadCount` equals `lastMessage.readCount`.
- `readCount` excludes the message sender.
- Hidden/deleted message body/content remains masked by existing message presenter behavior.

```json
  ]

```

Supported app-facing message types:

- `text`
- `image`
- `video`
- `file`
- `audio`
- `voice` as an app-facing alias mapped internally to audio behavior

Rules:

- Non-text media messages require at least one attachment.
- MIME type must match the message type.
- `allowAttachments`, `allowVideoMessages`, `allowVoiceMessages`, and `maxAttachmentSizeMb` are enforced where configured by Communication policy.
- Existing text send behavior is backward-compatible.
- `clientMessageId` idempotency is preserved; retries do not duplicate message attachments or notification generation.
- `replyToMessageId`, when supplied, must belong to the same conversation and pass existing visibility rules.

## Read Receipt And Message Info Contract

Read receipt rules:

- `readCount` excludes sender.
- Historical sender self-read rows are ignored.
- Sender self-read does not inflate counts.
- `participantsCount` includes active + muted participants and includes the sender/current actor where active or muted.
- `fullyRead = readCount + 1 >= participantsCount`.
- Frontend must set `readCount` from backend payloads and realtime events, not increment locally.
- `conversation.join` is subscribe-only and does not mark messages read.
- REST mark-read remains the source of truth.

Reader card shape:

```json
{
  "userId": "uuid",
  "user_id": "uuid",
  "displayName": "Ahmed Mohamed",
  "display_name": "Ahmed Mohamed",
  "userType": "teacher",
  "user_type": "teacher",
  "isMe": false,
  "is_me": false,
  "readAt": "ISO",
  "read_at": "ISO"
}
```

Teacher reader cards omit snake_case aliases.

Message info:

- Combines safe message preview plus readers.
- Sender is excluded from readers.
- Hidden/deleted body/content masking is preserved.
- This is not a delivery receipt system. No `deliveredAt`, delivery status, or double-grey checks exist in V1 Track A.

- Student: `teacher:<teacherUserId>`

Discovery rules:

- Parent contacts are teachers allocated to linked child classrooms. Role filters other than `teacher` return no contacts.
- Student contacts are teachers allocated to the current classroom. Role filters other than `teacher` return no contacts.
- Teacher contacts are allocated students and linked guardians for allocated students. Role filters support `student` and `parent`.
- Optional `q` filters only within already-authorized contacts.
- No global school directory search exists.
- No arbitrary user id lookup exists.

Create/reuse direct conversation request:

```json
{
  "contactId": "teacher:<userId>"
}
```

Rules:

- App routes resolve the contact through the authorized contact read model first.
- Clients never submit participant arrays.
- Existing active DIRECT conversations with exactly the actor and target participant are reused.
- If no active direct conversation exists, one is created with actor and target participants only.
- Initial message creation is deferred; create conversation does not send a message or emit message events.
- There is no database-level unordered pair uniqueness constraint; service-level lookup prevents duplicates under normal request flow.

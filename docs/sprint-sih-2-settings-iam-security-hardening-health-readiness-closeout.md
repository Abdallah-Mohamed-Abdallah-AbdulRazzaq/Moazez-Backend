## Summary Of Fixes

Sprint S-IH-2 closes the critical S-IH-1 blockers that could allow stale access after account disablement, overly broad email permissions, unsafe credential delivery retries, and health checks that only reflected database reachability.

Implemented:

- Disabled or suspended users are blocked by login, refresh, JWT auth, and scope resolution.
- Settings user disable/suspend now revokes active sessions.
- Login email identifiers are trimmed and lowercased before lookup and audit failure recording.
- `GET /api/v1/settings/permissions` now requires `settings.permissions.view`.
- Settings email connection, templates, deliveries, credential deliveries, and campaigns now use granular permissions.
- Credential delivery retries reuse one encrypted pending temporary password per recipient and apply credentials only after SMTP send succeeds.
- Health readiness now checks DB, Redis, storage, queue readiness, email readiness, and push readiness.
- Stale identity closeout e2e absent-route assertions were updated for notification, teacher messaging, and student reward routes that now intentionally exist.

## IAM Disabled-User And Session Invalidation Policy

Settings status updates now revoke sessions when the resulting user status is not `ACTIVE`.

Runtime enforcement points:

- Login continues to reject non-active users.
- Refresh loads the current user state, revokes sessions for non-active users, and rejects refresh rotation.
- JWT auth validates the session belongs to the token subject, reloads the user, revokes sessions for non-active users, and rejects stale access tokens.
- Scope resolution reloads the user and rejects non-active users before dashboard/app school scope can be used.
- Logout behavior remains unchanged.
- Change-password behavior remains unchanged; disabled users are blocked before reaching it by auth/scope guards.

## Credential Delivery Retry Safety Design

Credential delivery processing now uses a schema-free pending secret design:

1. Generate the temporary password once per delivery recipient.
2. Store only an encrypted pending temporary password in recipient metadata under internal key `pendingCredential`.
3. On retry, decrypt and reuse the same pending password instead of generating a new one.
4. Render and send the email with the same pending password.
5. Only after SMTP send succeeds, hash/apply the password, set `mustChangePassword=true`, update provisioning timestamps through the existing credential repository, revoke user sessions, and mark the recipient sent.
6. On SMTP failure, leave `User.passwordHash`, `mustChangePassword`, `passwordProvisionedAt`, and sessions unchanged.
7. Sent recipient metadata is replaced with sanitized provider counts and message id summary, so pending encrypted password is not returned by delivery history presenters after success.

No-leak behavior:

- Plain temporary passwords are not stored in database metadata.
- Pending encrypted temporary passwords are not exposed through presenters.
- Failure reasons redact temporary password patterns and email addresses.
- Direct credential generation endpoints keep their existing one-time reveal behavior.

Residual risk:

- If SMTP send succeeds but applying the credential or marking the recipient sent fails, the system cannot make provider delivery transactional without an outbox/provider acknowledgement redesign. The current implementation treats the recipient as failed and retries with the same pending password, avoiding password regeneration and avoiding an active unknown password after SMTP failure.

## Health Readiness Checks Added

`GET /api/v1/health` now reports sanitized readiness for:

- `db`: Prisma `SELECT 1`.
- `redis`: BullMQ Redis ping.
- `storage`: configured private/public bucket readiness through the storage service.
- `queues`: readiness/count summary for school email delivery, communication notifications, and communication push queues.
- `email`: global active school email connection readiness summary without provider secrets or school details.
- `push`: Firebase provider mode summary: disabled, dry-run, or send-enabled.

Health response behavior:

- Required dependency errors or degraded queue state degrade the overall health status.
- Optional unconfigured email/push providers report `skipped` instead of hard failure.
- Public messages are sanitized and do not include connection strings, bucket names, provider secrets, stack traces, SMTP passwords, or Firebase private key material.

## Security And No-Leak Summary

Confirmed hardening:

- Disabled users cannot keep using existing access or refresh sessions.
- Settings disable/suspend revokes active sessions.
- Refresh cannot issue new tokens for disabled/suspended users.
- Permission catalog route has explicit permission protection.
- Settings email routes no longer rely on broad `settings.security.*` permissions.
- Credential delivery retry does not regenerate a different temporary password for the same recipient.
- SMTP failure does not activate an unrevealed temporary password.
- Delivery presenter responses do not expose pending encrypted credential metadata or plaintext temporary passwords.
- Health output is sanitized.
